# v2 开发基础基线

这份文档定义 `aidianping v2` 的启动边界，避免新仓库一开始就重新走回“边做边改、边改边乱”的老路。

## 产品范围

v2 第一阶段只做用户侧三张主页面：

1. 首页 `/`
2. 工具列表页 `/tools`
3. 工具详情页 `/tools/[slug]`

本阶段支持：

- 普通搜索
- AI 搜索
- 分类、价格、平台、语言、排序筛选
- 详情查看
- 评论与评分

本阶段暂不做：

- 后台维护页面
- 运营配置台
- 复杂权限系统

## 仓库职责

### apps/web

负责：

- 页面渲染
- 路由
- 搜索工作台
- 列表与详情交互
- 评论与评分前端体验

### apps/api

负责：

- 工具列表接口
- 工具详情接口
- AI 搜索接口
- 评论评分接口
- mock 数据与未来真实数据源适配

### packages/contracts

负责：

- 前后端共享类型
- 列表、详情、AI 搜索、评论评分的数据契约

## 两人分工建议

### 开发者 A

负责：

- 产品交互细化
- 首页、列表页、详情页的前端骨架
- 搜索状态与 URL 同步
- 卡片、筛选条、评论区组件

### 开发者 B

负责：

- 数据模型
- mock 数据
- API 契约实现
- AI 搜索结果结构
- 评论与评分接口

## 启动顺序

严格按这个顺序推进：

1. 初始化 `apps/web`
2. 初始化 `apps/api`
3. 接入 `packages/contracts`
4. 准备 mock 数据
5. 先做静态页面骨架
6. 再接真实 API
7. 最后补测试和边界状态

## 第一周建议任务

### Day 1

- 初始化前端应用
- 初始化后端应用
- 接入共享 contracts
- 锁定首页、列表页、详情页信息架构

### Day 2

- 完成首页静态骨架
- 完成列表页静态骨架
- 输出工具卡片和筛选条组件规范

### Day 3

- 完成详情页静态骨架
- 输出评论与评分交互草案
- 后端完成 mock 数据结构

### Day 4

- 接列表接口
- 接详情接口
- 接 AI 搜索接口

### Day 5

- 接评论与评分接口
- 完成列表到详情的跳转闭环
- 补空态、错误态、加载态

## 质量要求

- 页面文案不得出现乱码
- 字段命名统一来自 contracts
- 页面状态优先由 URL 驱动
- 不把旧项目的页面和样式直接搬进来
- 每一阶段都保留可运行结果
