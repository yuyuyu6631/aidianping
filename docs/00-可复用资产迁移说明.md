# 可复用资产迁移说明

这份文档说明：从旧项目中，哪些资产被迁移进 `aidianping v2`，以及这些资产真实存在的价值。

## 已迁移资产

### 1. 共享领域模型与接口契约

来源价值：

- 旧项目已经沉淀出 AI 工具目录的核心实体边界
- 包括工具、分类、标签、评论、评分、榜单、场景、AI 搜索结果等概念
- 这些概念本身是稳定的，值得保留

本次在 `packages/contracts/src/index.ts` 中做了重新整理：

- `ToolSummary`
- `ToolDetail`
- `ToolTag`
- `ReviewPreview`
- `ReviewRecord`
- `ScoreRecord`
- `ScenarioSummary`
- `RankingSection`
- `ToolsListRequest`
- `ToolsListResponse`
- `AiSearchRequest`
- `AiSearchResponse`

实际价值：

- 为前后端提供统一数据语言
- 避免前端页面开发和后端接口命名各自漂移
- 为后续 mock 数据、API 设计、页面联调提供稳定基线

### 2. Monorepo 目录结构思路

来源价值：

- 旧项目已经证明：前端与共享契约拆分是合理的
- 共享契约包比把类型散落在页面目录里更利于迭代

本次迁移后形成的结构：

```text
apps/
  web/
  api/
packages/
  contracts/
docs/
```

实际价值：

- 方便两个人并行开发
- 前端和后端可以围绕同一份 contracts 对齐
- 后续后台、脚本、测试都可以继续扩展

### 3. 产品级数据边界思路

来源价值：

- 旧项目虽然 UI 实现不理想，但业务边界已经比较明确
- 列表页、详情页、AI 搜索、评论评分、榜单和场景是稳定需求

本次保留的不是旧代码，而是这些业务边界定义。

实际价值：

- 避免新仓库从零重新发明需求模型
- 可以直接进入“新设计 + 新实现”，而不是反复讨论最基础的数据边界

## 明确不迁移的资产

### 1. 旧首页、旧列表页、旧详情页实现

不迁移原因：

- 布局和交互已经背离新的产品方向
- 继续搬过去只会把旧包袱重新带进新仓库

### 2. 旧样式文件和页面壳层

不迁移原因：

- 旧样式中有大量为历史 UI 妥协的结构
- 不适合作为 v2 的设计系统基础

### 3. 旧设计稿文档

不迁移原因：

- 部分文档出现乱码和内容失真
- 文案准确性无法保证

### 4. 示例种子数据原文

不迁移原因：

- 旧仓库内部分示例数据已经存在乱码
- 字段结构可复用，但文本内容不适合直接迁移

## 当前结论

对于 `aidianping v2`，真正值得保留的是：

- 领域模型
- 接口契约
- 仓库结构
- 产品边界认知

真正应该放弃的是：

- 旧 UI 实现
- 旧样式层
- 旧页面组织方式
- 有乱码风险的文案与示例数据
