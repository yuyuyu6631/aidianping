# 05-前后端对接约束手册

## 1. 文档目标

本手册定义星点评 V1 项目前后端联调时必须遵守的固定约束。

目的不是解释业务，而是减少这几类问题：

- 字段反复变
- 接口格式不统一
- 登录态混乱
- 页面和接口各写各的
- 联调阶段频繁返工

---

## 2. 总体约束

- 以 [系统架构文档.md](系统架构文档.md) 为总技术基线。
- 以 [星点评 PRD（V1）.md](星点评%20PRD（V1）.md) 为功能范围基线。
- 超出 V1 范围的内容不允许混入当前接口设计。
- 任何字段、接口、状态流变更，必须先更新文档。

---

## 3. 接口契约约束

### 3.1 响应外层结构固定

除个别纯字典接口外，统一使用：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

约束：

- `code` 必须存在
- `message` 必须存在
- `data` 必须存在
- 不允许今天返回 `result`，明天返回 `data`

### 3.2 分页结构固定

列表接口统一返回：

```json
{
  "list": [],
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "total": 0,
    "totalPages": 0
  }
}
```

约束：

- 不允许某些接口用 `items`
- 不允许某些接口缺 `totalPages`
- 不允许前端自行计算分页总数

### 3.3 字段命名固定

统一使用 `camelCase`：

- `shortDesc`
- `primaryUsage`
- `officialUrl`
- `isFavorited`

禁止出现同义混用：

- `tool_name` 和 `toolName`
- `desc` 和 `shortDesc`
- `url` 和 `officialUrl`

---

## 4. 页面与接口边界约束

### 4.1 首页

首页只承接：

- 精选工具
- 专题模块
- 搜索入口

首页不承担：

- 完整复杂筛选
- 全量结果承载
- 后台编辑能力

### 4.2 工具列表页

列表页负责：

- 全站搜索
- 分类筛选
- 标签筛选
- 排序
- 分页

列表页不负责：

- 专题说明头图
- 专题专属文案

### 4.3 专题更多页

更多页是首页专题模块的二级聚合页，不是独立工具库。

必须满足：

- 有 `topic.slug`
- 有专题标题和说明
- 只展示该专题下工具

### 4.4 工具详情页

详情页必须包含：

- 工具基础信息
- 评分总览
- 评论列表
- 评论提交
- 同类推荐

### 4.5 后台

后台 V1 仅允许做：

- 工具 CRUD
- 分类 CRUD
- 标签 CRUD
- 专题 CRUD
- 上下架
- 精选配置
- 轻量评论隐藏

后台 V1 不允许扩展成：

- 多级审批流
- 外部投稿平台
- 复杂报表中心

---

## 5. 鉴权约束

### 5.1 Token 传递方式固定

所有登录态请求统一使用：

```http
Authorization: Bearer <accessToken>
```

不允许：

- 一部分接口从 header 取 token
- 一部分接口从 query 取 token
- 一部分接口从 body 取 token

### 5.2 刷新机制固定

- `accessToken` 过期走 `/api/auth/refresh`
- `refreshToken` 用于换新，不直接访问业务接口
- refresh 失败必须回登录态失效流程

### 5.3 后台权限固定

- 后台接口必须校验 `role=admin`
- 仅登录不等于可访问后台

---

## 6. 数据约束

### 6.1 Tool 状态位固定

工具状态仅允许：

- `draft`
- `published`
- `archived`

约束：

- 前台列表和详情只展示 `published`
- 后台可查看全部状态

### 6.2 评论约束固定

- 一个用户对同一工具只允许一条评论
- 新提交评论覆盖旧评论
- 评论状态仅允许 `visible` / `hidden`

### 6.3 专题约束固定

- 一个工具可属于多个专题
- 专题更多页按 `topic.slug` 查询
- slug 一旦发布，不允许随意修改

---

## 7. 前端实现约束

### 7.1 不直接依赖数据库模型

前端只能依赖：

- Swagger
- service 层类型定义
- 对接文档

不允许：

- 看到 Prisma 字段就直接拿来渲染
- 未经接口暴露就使用内部字段

### 7.2 页面不得直接拼接接口

必须通过 service 层统一请求。

不允许：

- 页面组件中写 `/api/tools?page=1`
- 多个页面各自复制请求逻辑

### 7.3 页面必须处理三态

每个核心页面必须处理：

- loading
- empty
- error

---

## 8. 后端实现约束

### 8.1 DTO 先行

新增或修改接口时，必须先明确：

- request DTO
- response DTO
- Swagger 注解

### 8.2 不允许静默变更契约

以下变更一律视为破坏性变更：

- 字段改名
- 字段类型变化
- 数组改对象
- 对象改数组
- 分页结构变化

### 8.3 后台状态影响前台展示

后端必须保证：

- `draft` 工具不会出现在前台
- `archived` 工具不会出现在前台
- `isFeatured=false` 不会进入首页精选

---

## 9. 错误处理约束

### 9.1 错误码语义固定

- `400` 参数错误
- `401` 未登录或 token 无效
- `403` 无权限
- `404` 资源不存在
- `409` 资源冲突
- `500` 服务内部错误

`409` 典型场景包括：

- 注册邮箱已被占用
- 重复收藏同一工具
- 创建分类/标签/专题时 slug 已存在

### 9.2 表单错误返回固定

校验错误统一返回：

```json
{
  "code": 400,
  "message": "参数校验失败",
  "errors": [
    { "field": "name", "message": "名称不能为空" }
  ]
}
```

前端按 `errors[].field` 做表单映射，不允许后端返回纯字符串拼接错误。

---

## 10. 联调发布约束

### 10.1 不允许边联调边改协议

联调期间如果必须改接口：

1. 先更新文档
2. 同步前端负责人
3. 再修改代码

### 10.2 提测前必须冻结接口

进入提测阶段后：

- 不再新增字段
- 不再修改分页结构
- 不再调整鉴权方式

### 10.3 Definition of Done

一个页面算真正完成，必须同时满足：

- UI 已完成
- 真实接口已接通
- 空态/错误态/加载态齐全
- 权限控制正确
- 数据刷新闭环正确

---

## 11. 禁止事项

- 禁止前端写死后端返回结构的“猜测字段”
- 禁止后端未经通知删除字段
- 禁止把后台临时字段暴露给前台正式页面
- 禁止在联调阶段继续使用本地假数据冒充真实接口
- 禁止用“先能跑再说”的方式破坏统一结构

---

## 12. 最终解释顺序

发生冲突时，优先级如下：

1. 当前文档约束
2. 系统架构文档
3. PRD
4. 代码实现

如果代码与文档冲突，以文档为准；如果文档明显错误，先改文档再改代码。
